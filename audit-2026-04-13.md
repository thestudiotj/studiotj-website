## Section 2 — Sorter-driven contract drift

### `collections` as plural array
**Status: Matches.** `portfolio.json` photos have `"collections": ["the-signature-collection", "atmospheric"]` (plural array, `portfolio.json:1353-1356`). The `Photo` interface in `lib/portfolio.ts:34` declares `collections: string[]`. The site reads this field but **never uses it for display logic** — collection membership is determined by the collection's own `photo_ids[]`, not by `photo.collections[]`. No drift. **Severity: no-impact.**

### `primary_collection` tiebreaker
**Status: Not present in JSON, not used by site.** `portfolio.json` has no `primary_collection` field on any photo. The site code has no reference to `primary_collection`. `sortCollections` in `lib/portfolio.ts:95-112` is collection-level, not photo-level. **Severity: no-impact.** If the sorter writes it later, the site will silently ignore it.

### Four locked collection slugs
**Status: Two slugs are wrong. Breaking when new JSON lands.**

| Target slug | Current JSON slug | Site file | Severity |
|---|---|---|---|
| `the-signature-collection` | `the-signature-collection` | `portfolio.json:9` | ✓ matches |
| `monochrome-moods` | `monochrome-moods` | `portfolio.json:9` | ✓ matches |
| `the-halcyon-collection` | `halcyon` | `portfolio.json:94` | **Breaking** — `/portfolio/halcyon` will 404 when slug is corrected |
| `the-atmospheric-collection` | `atmospheric` | `portfolio.json:125` | **Breaking** — `/portfolio/atmospheric` will 404 when slug is corrected |

The site generates routes from `generateStaticParams` (`portfolio/[slug]/page.tsx:11-13`) by reading whatever slugs are in `portfolio.json`. When the sorter outputs `the-halcyon-collection`, the old URL `/portfolio/halcyon` will produce a 404. There are no redirects configured in `next.config.js`.

### `stock_xmp_path` and mono-prep dual-write fields
**Status: Not present, not read.** Neither field appears in `portfolio.json` or in any site source file. **Severity: no-impact.**

### Crop fields
**Status: Not present, not read.** No crop field anywhere in `portfolio.json`, the `Photo` type (`lib/portfolio.ts:27-36`), or any component. **Severity: no-impact.**

### Thumbnail convention
**Status: Matches.** `Gallery.tsx:76` uses `photo.thumbnail_url`. Real R2 photos use `-thumb.jpg` suffix (`portfolio.json:1345`). The lightbox at `Lightbox.tsx:128` loads `photo.url` (full-res) only when not a `/sample/` path. Correct separation. **Severity: no-impact.**

### Photo URL stability
**Status: No individual photo URLs exist today — but collection URLs are fragile.**  
There are no `/portfolio/[slug]/[photoId]` routes. Photos are only accessible within collection pages. The current URL shape for browsing is `/portfolio/[collection-slug]` only.  
Implication: the `halcyon` → `the-halcyon-collection` slug rename will break existing URLs for that collection. A redirect must be in place before or at the same time the new JSON is deployed.  
For future rotating collections: since there are no per-photo URLs today, there is no URL stability issue for photos themselves. The constraint becomes relevant only when/if individual photo detail pages (`/portfolio/[slug]/[photoId]`) are added.

## Section 5 — Interaction matrix for deferred Website Evolution items

Three deferred items: **rotating/randomized collections**, **journal spark automation**, **SEO profile per collection**.

### Where each would touch current code

**Rotating/randomized collections:**
- `src/lib/portfolio.ts` — `sortCollections` function (lines 95-112): currently sorts by `sort_order` then by latest photo date. Randomization/rotation logic would go here or in a new function.
- `src/app/page.tsx:9-11` — the `sortCollections(...).slice(0, 4)` call: would need to decide which 4 collections surface.
- `src/components/CollectionCard.tsx` — no interface change needed, but will need real hero photo thumbnails loaded before rotating collections looks right (otherwise it's just rotating gradients).
- `public/portfolio.json` schema — would need some indicator of which collections are eligible for rotation vs always-pinned. The `sort_order` field already exists for this distinction.
- A cron/revalidation mechanism — not currently present anywhere.

**Journal spark automation:**
- `src/lib/blog.ts` — `getAllPosts` reads from `content/blog/*.mdx`. Automation would either write MDX files to that directory, or the site would need a new data source.
- `src/app/blog/page.tsx` — currently renders from the same `getAllPosts` function.
- Would likely require a new API route or Vercel cron job — nothing of this kind exists today.
- No sorter-side dependency; purely site-side.

**SEO profile per collection:**
- `src/lib/portfolio.ts` — `Collection` interface (lines 10-24): would add new fields (e.g. `seo_title`, `seo_description`, `og_image`).
- `src/app/portfolio/[slug]/page.tsx:16-23` — `generateMetadata` currently returns only `collection.name` and `collection.tagline`. SEO profile fields would be consumed here.
- `public/portfolio.json` — the sorter would need to write the new fields.

### Which Phase 1 changes are cheaper/safer before these

| Phase 1 change | Do before deferred item | Reason |
|---|---|---|
| Fix `halcyon` / `atmospheric` slugs + add redirects | Rotating collections | Rotating collections will reference these slugs in URLs; broken slugs mean broken rotation |
| Remove Winter Walk collection | Rotating collections | Reduces the collection pool to exactly 4, making rotation logic simpler and avoiding a non-canonical collection appearing in rotation |
| Homepage slice(0,4) fix | Rotating collections | Establishes the 4-collection display as the baseline before adding rotation logic on top |
| Add real hero thumbnails to CollectionCard | Rotating collections + SEO | Rotation with gradient placeholders provides no visual value; SEO profile is wasted without real imagery |
| About page | Journal spark / SEO | Google and Pinterest assess site completeness; sparse site undermines any SEO work |

### Code patterns that need to change shape before deferred items can land cleanly

**No per-photo URLs (affects rotating collections):** There are currently no individual photo detail pages. Rotating collections at the collection level is fine with the current URL shape. However, if the plan involves surfacing individual photos outside their original collection (e.g. a "random pick" on the homepage), the current `getCollectionPhotos(slug)` pattern (lib/portfolio.ts:76-85) ties photo access to collection membership. A photo-canonical lookup (`getPhoto(id)`) already exists at `lib/portfolio.ts:87-89` but has no associated route. This is not a blocker for collection-level rotation but would need resolving before photo-level rotation.

**`sortCollections` is the single sort function:** If rotating collections need a daily-seed random shuffle that persists across a 24-hour window, the current pure deterministic sort cannot do that. A rotation mechanism would need either a server-side daily seed or a `sort_order` override written by the sorter. Nothing in the current code supports time-based sorting.

**No revalidation strategy for `portfolio.json` changes:** The page at `/portfolio` is not `force-dynamic` and has no `revalidate` setting. Changes to `portfolio.json` require a redeploy to appear. Rotating collections would need either ISR or a rebuild trigger.
